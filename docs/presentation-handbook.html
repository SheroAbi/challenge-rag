Everlast RAG – Präsentations-Handbuch

Alles was du brauchst um das Projekt zu präsentieren, erklären und Fragen zu beantworten.

Was ist dieses Projekt? Eine RAG-Anwendung (Retrieval-Augmented Generation – ein System das erst relevante Dokumente sucht und dann eine KI-Antwort daraus generiert). Der User stellt eine Frage → die App sucht in der Datenbank nach passenden Inhalten → schickt diese als Kontext an Google Gemini → bekommt eine fundierte Antwort mit Quellenangaben zurück.

Flow in 20 Sekunden erklärt

(ähnlichste Chunks)

→

(im Chat anzeigen)

📖 Umsetzungsstory – Wie ich vorgegangen bin

Phase 1: Grundgerüst aufsetzen

Next.js 16 mit App Router initialisiert. Supabase-Projekt angelegt. Erste Migration: 001_initial_schema.sql mit pgvector-Extension, Basis-Tabellen. Seitenstruktur: Landing (/), About (/about), Knowledge (/knowledge). Layout mit Sidebar-Navigation, Dark/Light-Toggle.

Technische Details
  • App Router (Next.js Dateibasiertes Routing – jeder Ordner mit page.tsx wird automatisch eine URL)
  • pgvector Extension (PostgreSQL-Erweiterung die Vektoren speichern und Ähnlichkeitssuche kann)
  • Fonts: Manrope (Headlines), IBM Plex Mono (Code) via next/font/google
  • CSS Design-System mit oklch() Farbpalette für 40+ Custom Properties
Phase 2: Embedding-Versuch mit Google Gemini

Zuerst Google Gemini text-embedding-004 als Cloud-Embedding-Dienst angebunden. Problem: 429-Fehler (Quota-Limits) bei Batch-Verarbeitung vieler Dokumente, Timeouts bei großen Uploads. Obwohl ich dafür zahle, war die Zuverlässigkeit nicht gegeben.

Warum Cloud-Embeddings Probleme machten
  • 429 Too Many Requests (Server sagt: du sendest zu viele Anfragen pro Sekunde)
  • Jeder Batch von 100 Texten = 100 API-Calls → bei 5000 Lebensmitteln = 50 Batches = häufige Rate-Limit-Hits
  • Alternative: Batch-Endpoint → aber auch mit Limits und kein Streaming
  • Ergebnis: Upload bricht nach ~30% ab → unbrauchbar für Demos
Phase 3: Kurskorrektur → Lokale Embeddings mit Xenova

Lösung: @xenova/transformers mit all-MiniLM-L6-v2. Läuft komplett lokal auf der CPU, braucht keinen API-Key, hat keine Rate-Limits. Cold Start 2-5s (Modell laden), danach schnell.

Wie lokale Embeddings funktionieren
  • Transformer-Modell (neuronales Netz das Text in einen Zahlenvektor umwandelt – "versteht" die Bedeutung)
  • Mean Pooling + Normalisierung (Alle Token-Vektoren werden gemittelt und auf Einheitslänge gebracht – Standard für Ähnlichkeitsvergleich)
  • Modell wird beim Server-Start heruntergeladen (~22 MB), dann gecached
  • preloadModel() auf Modulebene → Eager Initialization (Modell wird sofort beim Import geladen, nicht erst beim ersten Aufruf)
Phase 4: Multi-Theme + Hybrid-Retrieval

Theme-System eingeführt: jedes Thema (food, saas_docs, exercises) bekommt eigene DB-Tabelle + RPC + Prompt-Hints. Food zusätzlich: Hybrid-Retrieval (zwei Suchstrategien gleichzeitig: Textsuche UND Vektorsuche, Ergebnisse zusammenführen für bessere Trefferquote).

Hybrid vs. Semantic Retrieval
  • Semantic-only (SaaS, Exercises): Frage → Embedding → pgvector Cosine Distance → Top-K Treffer
  • Hybrid (Food): Promise.all(lexical, semantic) parallel → mergeAndRerank()
  • Lexical Search (Textbasierte Suche mit pg_trgm – findet auch Tippfehler und Teilwörter wie "Hähnch" → "Hähnchenbrust")
  • Beide Ergebnisse werden dedupliziert und nach Score sortiert
Phase 5: Upload-Pipeline + Knowledge Dashboard

SSE-basierter Upload mit Echtzeit-Fortschritt. Zwei Pfade: Food-TSV (spezieller MFP-Parser) und Universal (JSON/JSONL/CSV/TSV). Knowledge-Dashboard mit Live-Stats aus Supabase.

SSE-Streaming erklärt
  • SSE (Server-Sent Events) (Einweg-Datenstrom vom Server zum Browser – Server schickt laufend Updates, Browser muss nur zuhören)
  • Events: init → parsing → embedding_progress (X/Y) → indexing → complete
  • Im Browser: ReadableStream liest den Datenstrom und parsed data: Zeilen
  • Vorteil gegenüber WebSocket: einfacher, läuft durch Proxies, HTTP/2 kompatibel

Warum diese Schichtung?
  • Separation of Concerns (Jede Schicht hat genau eine Aufgabe – UI kümmert sich um Darstellung, Service um Logik, AI um Modelle)
  • Provider Pattern (Interface + Implementierung getrennt – man kann den Embedding-Anbieter austauschen ohne den Rest zu ändern)
  • Single Responsibility (Jede Datei macht genau eine Sache – z.B. parser.ts parsed nur, indexer.ts indexiert nur)

🔄 End-to-End Flows

Flow 1: User stellt eine Frage (Chat Query)
  1. Frontend (animated-ai-chat.tsx): User tippt Frage, wählt Theme (z.B. "food"), klickt Senden
  2. API (/api/chat/query): POST mit {question, theme, topK} → Zod-Validierung → ragOrchestrator.query()
  3. Orchestrator (rag-orchestrator.ts): Entscheidet Retrieval-Strategie basierend auf Theme
  4. Retriever (retriever.ts): Embedding der Frage → Supabase RPC match_rag_*_chunks → Top-K Treffer
  5. Kontext: Treffer → contextChunks[] mit content, documentTitle, chunkId
  6. Generator (gemini-answer-generator.ts): System-Prompt + Kontext + Frage → Gemini REST API → Antwort
  7. Response: {answer, citations[], resultTable, latencyMs, model} → Frontend rendert Markdown + Quellen-Cards
Flow 2: Datei-Upload (SSE Streaming)
  1. Frontend (rag-upload-modal.tsx): Drag&Drop Datei → Tabelle auswählen → Upload-Button
  2. API (/api/knowledge/upload): FormData empfangen → import_runs anlegen → SSE-Stream starten
  3. Parser (parser.ts): Format erkennen (JSON/JSONL/CSV/TSV) → Zeilen/Records parsen → Typisierung
  4. Indexer (indexer.ts): Batch-Embedding (100er-Batches) → Upsert in DB-Tabelle
  5. SSE-Events: init → parsing → embedding_progress (34/100) → indexing → complete
  6. Frontend: ReadableStream parsed SSE-Events → Progressbar + Terminal-Log + Stats
Flow 3: Neues Theme hinzufügen (Erweiterbarkeit)
  1. SQL-Migration: Neue Tabelle rag_NEW_chunks + match_rag_NEW_chunks RPC + HNSW-Index
  2. Theme-Registry: In theme-router.ts neuen Eintrag: { tableName, matchRpc, promptHint, emptyState }
  3. Zod-Schema: In schemas.ts den themeKeySchema um den neuen Key erweitern
  4. Fertig! Upload, Retrieval und Chat funktionieren automatisch über die bestehende Pipeline

→ ~20 Zeilen Code + 1 SQL-Migration für ein neues Theme. Das zeigt die Erweiterbarkeit.

src/server/services/rag-orchestrator.ts Service

RAG-Orchestrator (Herzstück, ~130 Zeilen) – Koordiniert die gesamte Query-Pipeline: Theme auflösen → Retriever aufrufen → Kontext zusammenbauen → Antwort generieren. Fassaden-Pattern (eine einzelne Schnittstelle die mehrere komplexe Subsysteme vereinfacht – "ein Hauptschalter für alles").

Universal Retriever (~150 Zeilen) – Embedding der Frage → Supabase RPC aufrufen → Ergebnisse normalisieren. Unterstützt sowohl Semantic Search (Bedeutungsbasierte Suche mit Vektoren) als auch Hybrid Search (Vektor + Text kombiniert).

Code-Level Details

Universal Indexer (~120 Zeilen) – Nimmt geparste Records entgegen → Batch-Embedding (Texte in 100er-Gruppen embedden statt einzeln – viel schneller) → Upsert (Update-or-Insert: wenn der Datensatz existiert wird er aktualisiert, sonst neu angelegt) in Supabase.

💬 Interview Q&A – Spickzettel

15 Fragen die kommen könnten – mit fertigen Antwortbausteinen.

F1: Welches State Management nutzt du?

Antwort

React useState + useCallback – für diese App reicht Reacts eingebautes State Management. Es gibt keinen globalen State (kein Redux, kein Zustand). Jede Komponente verwaltet ihren eigenen State: Chat-Nachrichten in animated-ai-chat.tsx, Upload-Stage in rag-upload-modal.tsx. Das Theme wird über next-themes verwaltet (Context-basiert). Wenn die App wachsen würde, wäre Zustand mein Favorit – leichtgewichtig, wenig Boilerplate, gut mit React 19.

F2: Welche Tests hast du?

Antwort

Aktuell: keine automatisierten Tests – bewusste Entscheidung für die Challenge-Zeitschiene. Bei produktivem Code würde ich: 1) Unit-Tests mit Vitest für Parser, Normalizer, Schemas. 2) Integration-Tests für API-Routes mit supertest. 3) E2E-Tests mit Playwright für den Upload-Flow und Chat. Die Architektur ist testbar (Dependency Injection, Interfaces, reine Funktionen).

F3: Warum Xenova statt Cloud-Embeddings?

Antwort

Zuerst Google Gemini text-embedding-004 probiert → 429 Rate-Limits beim Batch-Upload (obwohl bezahlt). Für eine Demo braucht man Zuverlässigkeit. Xenova (all-MiniLM-L6-v2): Lokal, kostenlos, keine Limits, ~20ms/Text. Trade-off: 384d statt 768d → etwas weniger Qualität, aber für Retrieval ausreichend. Cold Start ~3s beim ersten Laden.

F4: Warum Gemini und nicht OpenAI?

Antwort

1) Gemini 2.0 Flash ist schnell und günstig. 2) Kostenlose Tier beim Start. 3) Gute Qualität für RAG. 4) Aber: dank Provider Pattern kann ich jederzeit zu OpenAI wechseln – nur eine neue Klasse die AnswerGenerator Interface implementiert. Architektur-Entscheidung: welcher Provider ist austauschbar, die Pipeline bleibt gleich.

F5: Warum SSE statt WebSocket für Upload?

Antwort

SSE ist unidirektional (Server → Client). Für Upload-Fortschritt brauche ich nur eine Richtung. WebSocket wäre Overkill: bidirektional, Heartbeats, Reconnect-Logik. SSE: einfacher, HTTP/2-kompatibel, läuft durch Proxies, automatischer Reconnect im Browser. Ein paar Zeilen Code vs. eine ganze Library.

F6: Warum Supabase statt eine eigene DB?

Antwort

Supabase = PostgreSQL + pgvector + Auth + RPC + Dashboard out of the box. Kein DB-Setup, kein Hosting. pgvector drin → brauche keine separate Vektordatenbank (Pinecone, Weaviate). Managed Infrastructure. Free Tier für Demos. Supabase JS-Client hat gute TypeScript-Typen. Migrationen per SQL-Dateien – versionierbar und reproduzierbar.

F7: Wie skaliert das System?

Antwort

Horizontal: Next.js auf Vercel = auto-scaling. DB: Supabase Pro hat Connection Pools + Read Replicas. Embedding: Cold Start mit Model Cache → warm nach 1. Request. HNSW-Indizes: sublinear O(log n) statt linear scan. Bottleneck: CPU-basierte Embeddings – bei hoher Last: GPU oder Cloud-Embeddings nutzen. Theme-System: neue Themes ohne Code-Änderung.

F8: Was würdest du anders machen?

Antwort

1) Token-Streaming für LLM-Antworten (SSE aus Gemini → sofort anzeigen). 2) Chat-History Persistence in DB speichern. 3) Error Boundaries in React. 4) Tests von Anfang an. 5) Caching Layer für wiederholte Fragen (Redis). 6) Embedding Queue mit Bull/BullMQ für große Uploads. 7) Multi-Tenant von Anfang an richtig machen.

F9: Was ist das Provider Pattern?

Antwort

Ein Interface (Vertrag) definiert welche Methoden existieren müssen. Eine Implementierung (Klasse/Objekt) sagt wie. Beispiel: EmbeddingProvider hat embed() + embedBatch(). LocalEmbeddingProvider implementiert mit Xenova. Morgen kann ich OpenAIEmbeddingProvider schreiben ohne den Orchestrator anzufassen. Der Orchestrator kennt nur das Interface.

F10: Hybrid vs. Semantic Retrieval?

Antwort

Semantic: Frage → Vektor → Cosine Similarity → Top-K. Gut für bedeutungsähnliche Texte. Hybrid: Semantic + Lexical (pg_trgm) parallel → Ergebnisse mergen. Lexical fängt ab was Vektoren verpassen: exakte Matches, Tippfehler ("Hähnch" → "Hähnchenbrust"). Hybrid nur für Food aktiviert weil dort Produktnamen exakt gesucht werden. SaaS/Exercises → nur Semantic.

F11: Welche Design Patterns nutzt du?

Antwort
  • Provider Pattern – austauschbare AI-Implementierungen
  • Facade/Orchestrator – eine Schnittstelle für die gesamte Pipeline
  • Registry Pattern – zentrale Theme-Verwaltung
  • Strategy Pattern – verschiedene Parser für verschiedene Formate
  • State Machine – Upload-Stages (idle→uploading→complete)
  • Singleton – Orchestrator + Embedding Pipeline
  • Error Hierarchy – AppError → ValidationError/NotFoundError

F12: Wie geht ihr mit Fehlern um?

Antwort

API-Layer: try/catch → apiError() → konsistentes JSON mit requestId, HTTP-Statuscode, Error-Code. Frontend: Graceful Degradation – Fallback-Werte bei API-Fehler. Upload: Einzelne Fehler werden geloggt (import_errors), Upload läuft weiter. Zod: safeParse → Validation-Fehler werden als 400er zurückgegeben.

F13: Warum TypeScript mit strict mode?

Antwort

Strict mode fängt Bugs vor der Laufzeit ab: null-Pointer, fehlende Properties, falsche Typen. Zusammen mit Zod (Runtime) → doppelte Absicherung: TypeScript prüft zur Compile-Zeit, Zod zur Laufzeit. z.infer synchronisiert beides automatisch. Kosten: etwas mehr Tipparbeit. Nutzen: weniger Bugs in Production.

F14: Was passiert wenn Gemini ausfällt?

Antwort

Retrieval funktioniert trotzdem (läuft lokal + Supabase). Nur die Antwort-Generierung fällt aus → Error-Response an den User: "Antwort konnte nicht generiert werden". Nächster Schritt wäre Fallback-Provider: wenn Gemini 5xx → OpenAI versuchen. Dank Interface: providers = [gemini, openai] → erster der antwortet gewinnt.

F15: Nächste Schritte?

Antwort
  1. Token-Streaming (SSE von Gemini → wortweise Anzeige)
  2. Chat-Verlauf in DB persistieren
  3. React Error Boundary einbauen
  4. Unit + Integration Tests mit Vitest
  5. Caching-Layer für häufige Fragen
  6. Dokumentenanalyse: PDF/Word Upload
  7. Multi-User Auth mit Supabase Auth

🤖 KI & Machine Learning

BegriffWas es ist, einfach erklärt
RAGRetrieval-Augmented Generation – die KI sucht erst in Dokumenten nach Infos und baut die Antwort dann daraus. Wie ein Schüler der erst im Buch nachschlägt bevor er antwortet.
EmbeddingText wird in eine Liste von Zahlen (Vektor) umgewandelt die die Bedeutung darstellt. "Hund" und "Welpe" bekommen ähnliche Zahlen. Populäre Modelle: all-MiniLM-L6-v2 (lokal), text-embedding-3-small (OpenAI), text-embedding-004 (Google).
VektorEine geordnete Liste von Zahlen, z.B. [0.12, -0.85, 0.33, ...] mit 384 Einträgen. Stell dir einen Punkt in einem 384-dimensionalen Raum vor – je näher zwei Punkte beieinander liegen, desto ähnlicher sind die Texte.
Cosine SimilarityMaß dafür, wie ähnlich zwei Vektoren sind (1.0 = identisch, 0.0 = komplett verschieden). Findet heraus ob "günstiges Essen" und "preiswerte Mahlzeit" ähnlich sind – ja, ~0.85.
ChunkingGroße Dokumente werden in kleine Stücke geschnitten (z.B. 500 Zeichen) damit die KI sie besser verarbeiten kann. Wie ein Buch in Karteikarten aufteilen.
LLMLarge Language Model – eine riesige KI die auf Milliarden von Texten trainiert wurde. Beispiele: GPT-4 (OpenAI), Gemini (Google), Claude (Anthropic), LLaMA (Meta). Kann Texte verstehen, zusammenfassen, beantworten, übersetzen.
PromptDie Anweisung die man der KI gibt. Ein guter Prompt = gute Antwort. Beispiel: "Du bist ein Ernährungsexperte. Beantworte basierend auf folgendem Kontext: [Chunks]"
TokenDie kleinste Einheit die eine KI verarbeitet – meistens ein Wort oder Wortteil. "Hähnchenbrust" ≈ 3 Tokens. GPT-4 kann ~128.000 Tokens pro Anfrage verarbeiten.
TransformerDie Architektur hinter allen modernen KI-Modellen (seit 2017). Nutzt "Attention" um zu verstehen welche Wörter in einem Satz zusammengehören. Library: @xenova/transformers (JS), transformers (Python, Hugging Face).
ONNX RuntimeLaufzeitumgebung die KI-Modelle effizient ausführt. Macht es möglich, ein Python-trainiertes Modell in JavaScript laufen zu lassen. Library: onnxruntime-node.
Mean PoolingAus vielen Token-Vektoren einen einzigen Durchschnittsvektor berechnen. Wie das Durchschnittsgesicht einer Gruppe – repräsentiert alle.
Normalisierung (Vektoren)Vektor auf Länge 1 skalieren damit Cosine Similarity korrekt funktioniert. Wie bei Noten: Erst normalisieren, dann vergleichen.
Batch ProcessingStatt Texte einzeln zu embedden, werden 100 auf einmal verarbeitet. 100× schneller als einzeln. Wie Wäsche: alles zusammen in die Maschine statt jedes Kleidungsstück einzeln waschen.

🌐 Web-Architektur & Backend

BegriffWas es ist, einfach erklärt
APIApplication Programming Interface – eine Schnittstelle über die Programme miteinander reden. Wie ein Kellner: du bestellst (Request), er bringt das Essen (Response). Wir nutzen: REST API über Next.js Route Handlers.
RESTArchitekturstil für APIs: jede URL ist eine Ressource, HTTP-Methoden (GET/POST/PUT/DELETE) sagen was passieren soll. Standard in ~90% aller Web-APIs.
Route HandlerIn Next.js: eine Datei (route.ts) die HTTP-Anfragen direkt bearbeitet. export async function POST(req) = diese Datei handelt POST-Requests. Kein Express.js nötig.
SSE (Server-Sent Events)Einweg-Datenstrom vom Server zum Browser. Perfekt für Fortschrittsanzeigen (Upload: "10/100 fertig... 50/100... done!"). Einfacher als WebSocket. Browser hat eingebauten EventSource.
WebSocketBidirektionaler Echtzeit-Kanal zwischen Server und Browser. Für Chat, Multiplayer-Games, Live-Daten. Libraries: socket.io (150M+ Downloads/Monat), ws.
MiddlewareCode der zwischen Anfrage und Antwort läuft – z.B. Auth-Check, Logging, Rate-Limiting. Wie ein Türsteher der prüft bevor du reinkommst. Libraries: express, Next.js Middleware.
RPCRemote Procedure Call – eine Funktion die über Netzwerk aufgerufen wird als wäre sie lokal. In Supabase: SQL-Funktionen die per supabase.rpc('match_food_items', {...}) aufgerufen werden.
FormDataBrowser-API um Dateien + Textfelder zusammen an den Server zu senden. Standard für File-Uploads. Jeder Browser unterstützt das seit ~2012.
Environment VariablesGeheime Einstellungen die nicht im Code stehen (API-Keys, DB-URLs). Liegen in .env.local. NEXT_PUBLIC_ = öffentlich sichtbar, ohne Prefix = nur Server.
Edge FunctionServerless-Funktion die an vielen Standorten weltweit läuft (CDN Edge). Schnell weil nahe am User. Supabase und Vercel bieten diese an. Library: Deno Runtime.

⚛️ React & Frontend

BegriffWas es ist, einfach erklärt
ReactDie populärste UI-Library (von Meta). Baut Benutzeroberflächen aus wiederverwendbaren Komponenten. ~23M Downloads/Woche auf npm. Konkurrenten: Vue.js, Svelte, Angular.
Next.jsReact-Framework das Server-Rendering, Routing, API-Routes und Optimierungen mitbringt. Von Vercel. ~7M Downloads/Woche. Alternativen: Remix, Nuxt (Vue), SvelteKit.

useCallbackReact Hook der eine Funktion cacht. Ohne useCallback wird die Funktion bei jedem Render neu erstellt → unnötige Re-Renders bei Kind-Komponenten.
useTransitionReact 18+ Hook: markiert eine Aktion als "nicht dringend" damit die UI reaktionsfähig bleibt. Perfekt für API-Calls: User kann weitertippen während die Antwort lädt.
Framer MotionDie populärste React-Animations-Library. <motion.div animate={{opacity:1}}> statt manuelles CSS. ~5M Downloads/Woche. Macht komplexe Animationen simpel.
Tailwind CSSUtility-first CSS Framework. Statt .btn { background: blue } schreibst du class="bg-blue-500". ~10M Downloads/Woche. Wir nutzen Tailwind 4 mit @theme Blocks.
next-themesLibrary für Dark/Light Mode in Next.js. 3 Zeilen Code für perfektes Theme-Switching mit System-Erkennung und Flash-Vermeidung.
react-markdownLibrary die Markdown-Text als React-Komponenten rendert. Mit remark-gfm Plugin auch: Tabellen, Checkboxen, Strikethrough.
Lucide IconsIcon-Library mit 1400+ sauberen SVG-Icons als React-Komponenten. Fork von Feather Icons. <Search size={16} />.

🗄️ Datenbank & Supabase

BegriffWas es ist, einfach erklärt
PostgreSQLDie mächtigste Open-Source-Datenbank der Welt. Kann alles: relationale Daten, JSON, Volltextsuche, Vektoren. Basis von Supabase. Konkurrent: MySQL, aber Postgres gilt als technisch überlegen.
Supabase"Open Source Firebase Alternative". Managed PostgreSQL + Auth + Echtzeit + Storage + Edge Functions. Kostenloser Tier für Demos. Client-Library: @supabase/supabase-js.
pgvectorPostgreSQL-Extension für Vektorspeicher. Statt eine separate Vektordatenbank (Pinecone $$$, Weaviate) zu betreiben, einfach CREATE EXTENSION vector – alles in einer DB.
pg_trgmPostgreSQL-Extension für Trigram-basierte Textsuche. Findet "Hänchenbrust" auch wenn der User "Hänchenb" tippt. Wie autocomplete, aber in der Datenbank.
HNSW-IndexHierarchical Navigable Small World – moderner Vektorindex. Findet die ähnlichsten Vektoren in O(log n) statt O(n). Wie ein Navi: statt alle Straßen abzulaufen, navigiert es intelligent. Schneller als IVFFlat für <1M Vektoren.
GIN-IndexGeneralized Inverted Index – PostgreSQL-Index für Volltext- und Trigram-Suche. Dreht die Suche um: statt "in welchem Dokument steht X" → Index speichert "X steht in Dok 1, 5, 99".
MigrationEine SQL-Datei die die Datenbankstruktur verändert (Tabelle anlegen, Spalte hinzufügen). Wird versioniert und in Reihenfolge ausgeführt. Wie Git-Commits, aber für die Datenbank.
UpsertUPDATE + INSERT in einem Schritt: Wenn der Datensatz existiert → aktualisieren. Wenn nicht → neu anlegen. Spart eine Abfrage. supabase.from('table').upsert(data, {onConflict: 'id'}).
RLS (Row Level Security)Supabase/PostgreSQL-Feature: Zugriff wird auf Zeilenebene kontrolliert. "User A darf nur seine eigenen Daten sehen." Wird per SQL-Policy definiert.
Service Role KeyPrivilegierter API-Key der RLS umgeht. Nur serverseitig nutzen! Wer diesen Key hat, hat vollen DB-Zugriff. Liegt in SUPABASE_SERVICE_ROLE_KEY.

🏗️ Design Patterns & Architektur

BegriffWas es ist, einfach erklärt
Provider PatternEin Interface definiert den Vertrag ("was"), verschiedene Klassen implementieren ("wie"). Wie ein USB-Anschluss: egal ob Maus oder Tastatur – der Anschluss ist gleich. Unser Beispiel: EmbeddingProvider → Xenova oder OpenAI.
SingletonEs existiert nur ein einziges Objekt einer Klasse im gesamten Programm. Unser ragOrchestrator und die Embedding-Pipeline sind Singletons – spart Speicher, verhindert Inkonsistenzen.
Facade PatternEine einfache Schnittstelle die etwas Komplexes verbirgt. Der ragOrchestrator.query() ist eine Fassade: ein Aufruf, aber dahinter laufen Retriever, Generator, Normalizer.
Registry PatternZentrale Map die Objekte nach Key verwaltet. themeConfigs['food'] → sofort die richtige Config. Neues Theme hinzufügen = ein neuer Eintrag in der Map.
Strategy PatternVerschiedene Algorithmen austauschbar machen. Unser Parser: .json → JSON-Strategie, .csv → CSV-Strategie. Gleiches Interface, unterschiedliche Implementierung.
State MachineDefinierte Zustände mit erlaubten Übergängen. Upload-Modal: idle → selected → uploading → processing → complete. Verhindert unmögliche Zustände (z.B. "processing" ohne vorheriges "uploading").
Dependency Injection (DI)Abhängigkeiten von außen reingeben statt im Code fest verdrahten. new Orchestrator(retriever, generator) statt new Orchestrator() mit fest verdrahtetem Retriever. Macht Testen und Austauschen trivial.
Separation of ConcernsJedes Modul hat genau eine Aufgabe. UI-Code kennt keine DB-Queries. API-Routes kennen keine Rendering-Logik. Wie in einer Küche: Koch, Kellner und Kassierer haben verschiedene Jobs.
Graceful DegradationWenn etwas schief geht, funktioniert die App trotzdem – nur eingeschränkt. Statt Crash → Fallback-Werte anzeigen. Knowledge-Dashboard: API-Fehler → "0 Einträge" statt weiße Seite.
Fail-FastFehler sofort beim Start melden, nicht erst wenn es zu spät ist. Gemini-Config: kein API-Key? → Sofort Error beim Import statt mysteriöser Fehler 10 Minuten später bei einem User-Request.
Cold StartDie erste Anfrage ist langsam weil etwas geladen werden muss (z.B. KI-Modell). Danach schnell. Wie ein Auto im Winter: erst Motor warmfahren, dann normal. Unser Embedding-Modell: ~3s → danach <50ms.

⚠️ Known Gaps – Ehrlich dokumentiert

1. Keine automatisierten Tests

Kein Unit/Integration/E2E Testing. Architecture ist testbar (DI, Interfaces), Tests würde ich zuerst für Parser + Schemas schreiben.

2. Kein Token-Streaming

Gemini-Antwort kommt als ganzer Block statt wortweise. User wartet 2-5s ohne Feedback. Fix: SSE-Stream aus der Gemini API.

3. Kein Chat-Verlauf

Nachrichten nur im React-State (RAM). Page-Reload = alles weg. Fix: chat_messages Tabelle in Supabase.

4. Keine Error Boundary

Frontend-Fehler → weißer Bildschirm. Fix: React <ErrorBoundary> Komponente.

5. Cold Start Latenz

Erster Embedding-Call ~2-5s (Modell laden). Danach <50ms. Workaround: preloadModel() existiert bereits.

6. Single-Tenant

Kein Multi-User Support. Service Role Key direkt genutzt. Für Production: Supabase Auth + RLS Policies.

7. Kein Rate Limiting

API-Endpunkte ungeschützt. Fix: Middleware mit Token-Bucket oder Vercel Edge Middleware.

8. Keine Monitoring/Logging

Nur console.log. Fix: Structured Logging (Pino) + Error Tracking (Sentry) + APM.
